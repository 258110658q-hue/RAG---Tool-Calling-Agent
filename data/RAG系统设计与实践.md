# RAG 系统设计与实践指南

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与文本生成相结合的技术架构。它的核心思想是：在 LLM 生成回答之前，先从外部知识库中检索相关信息，然后将检索结果作为上下文提供给 LLM，使其能够基于"事实"而非"记忆"来回答问题。

## RAG 的核心流程

一个标准的 RAG 系统包含以下步骤：

1. **文档解析（Parsing）**：将 PDF、Markdown、HTML 等格式的文档转换为纯文本
2. **文本分块（Chunking）**：将长文档切分成适当大小的小块
3. **向量化（Embedding）**：使用 Embedding 模型将文本块转换为向量
4. **向量存储（Indexing）**：将向量存入向量数据库
5. **检索（Retrieval）**：用户查询 → 查询向量化 → 相似度搜索 → 返回 Top-K 结果
6. **生成（Generation）**：将检索结果 + 用户问题一起发送给 LLM，生成最终回答

## RAG 需要向量数据库的根本原因

LLM 有两大根本性限制：

### 1. 知识截止日期
LLM 的训练数据有截止日期（如 2023 年初），无法知道之后发生的事情。RAG 通过实时检索外部文档解决了这个问题——你可以随时更新知识库，Agent 就能获取最新信息。

### 2. 上下文窗口容量
即使是最先进的模型，上下文窗口也有限（128K-1M token）。这意味着：
- 不能把所有文档都放进 Prompt
- 长上下文会导致注意力分散（"Lost in the Middle" 问题）
- Token 成本随上下文长度线性增长

向量数据库作为"外部记忆"，按需检索最相关的内容，只把真正有用的信息放进上下文窗口。

## 分块策略的深度对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 固定大小分块 | 按固定字符/token数切分 | 实现简单，速度最快 | 可能切断句子或段落 | 结构统一的文档 |
| 递归分块 | 按段落→句子→字符优先级逐级切分 | 保持语义完整性 | 实现复杂，速度中等 | 结构化文档（Markdown） |
| 语义分块 | 用模型判断语义边界 | 效果最好 | 速度最慢，成本最高 | 对质量要求极高的场景 |
| 句子级分块 | 按句子边界切分 | 粒度细，不切断句子 | 可能丢失上下文 | 问答系统 |

### 分块大小的选择

- **太小（<100 token）**：Chunk 只包含孤立信息，缺乏上下文，LLM 难以理解
- **太大（>2000 token）**：包含过多无关信息，稀释了相关内容的信号
- **推荐范围**：256-1024 token（中文约 200-800 字符）

### Overlap 的作用

相邻 Chunk 之间保留 10%-20% 的重叠文本。作用是：
- 防止关键信息恰好落在分块边界上被切断
- 让 LLM 在阅读多个 Chunk 时有连续的上下文感

## 检索质量优化

### Rerank（重排序）

基础检索（如余弦相似度）返回的 Top-K 结果可能不是最优排序。Rerank 模型会对初检结果进行二次精排：
1. 初检：向量搜索返回 Top-20（速度快，精度低）
2. Rerank：用 Cross-Encoder 模型对 20 个结果重新打分排序
3. 取 Rerank 后的 Top-5 送入 LLM

Rerank 显著提升检索精度，代价是增加一次模型调用。

### 混合检索（Hybrid Search）

结合稀疏检索（BM25 关键词匹配）和稠密检索（向量语义匹配）的优势：
- 向量检索擅长语义相似（"怎么退款" ↔ "退货政策"）
- 关键词检索擅长精确匹配（产品型号、编号、专有名词）
- 混合检索对两者结果进行加权融合

## RAG 的常见问题与对策

### 问题 1：检索到不相关的内容
- **对策**：设置相似度阈值，低于阈值的结果直接丢弃
- **对策**：优化 Embedding 模型选择（通用领域 vs 特定领域）

### 问题 2：检索到过时/矛盾的信息
- **对策**：文档版本管理，定期更新索引
- **对策**：元数据过滤（按时间、来源过滤）

### 问题 3：多跳推理失败
- **对策**：Agent 模式——让 LLM 自主决定是否需要二次检索
- **对策**：知识图谱辅助——结构化知识补充非结构化文档

## RAG vs 微调

| 维度 | RAG | 微调 (Fine-tuning) |
|------|-----|-------------------|
| 知识更新 | 实时，更新文档即可 | 需重新训练 |
| 可解释性 | 高，可追溯到具体文档 | 低，知识融入模型参数 |
| 成本 | 检索 + 推理成本 | 训练 + 推理成本 |
| 幻觉控制 | 较好，答案基于检索内容 | 较差，模型可能编造 |
| 适用场景 | 需要实时知识、可溯源的场景 | 需要改变模型行为风格的场景 |

两者不是互斥的——许多生产系统同时使用 RAG 和微调。

## 总结

RAG 是目前最主流的知识增强方案。它用"检索 + 生成"的方式弥补了 LLM 在知识时效性和上下文容量上的短板。一个高质量的 RAG 系统需要在分块策略、Embedding 模型、检索算法和 Prompt 设计等多个环节精心调优。
